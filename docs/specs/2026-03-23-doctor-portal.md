# Портал для врачей (Лендинг результатов сессий)

## Executive Summary
Разработка отдельного защищенного веб-интерфейса (лендинга) по адресу `/doctors` для врачей. Портал предоставляет удобный табличный вид завершенных сессий пациентов с возможностью фильтрации по дате и ФИО врача, а также быстрым доступом к просмотру и скачиванию PDF-отчетов. Интегрирован с Битрикс24 для синхронизации привязки пациентов к врачам.

## Problem Statement
Врачам необходим быстрый, изолированный и визуально понятный доступ к отчетам (PDF) по пройденным опросникам их пациентов. Использование общей админ-панели неудобно для их повседневной работы. Новый портал решает эту проблему, предлагая сфокусированный интерфейс с автосохраняемыми фильтрами и автоматическим подтягиванием данных из нужных воронок Битрикс24.

## Success Criteria
1. Портал доступен по адресу `https://opros-izdorov.ru/doctors`.
2. В существующей SQLAdmin панели появилась вкладка «Настройка врачей» для управления учетными записями.
3. Данные о враче для каждой сессии корректно извлекаются из Битрикс24 в зависимости от воронки сделки.
4. Врачи могут авторизоваться и видеть таблицу сессий со всем требуемым функционалом (ФИО, время, длительность, PDF).
5. Состояние фильтров (особенно по ФИО врача) сохраняется на стороне клиента (например, в localStorage).

## User Personas
- **Врач (Медицинский специалист)**: Цель — быстро найти сессию пациента за выбранный период и открыть/скачать PDF-отчет для приема.
- **Администратор клиники**: Цель — завести или отключить учетную запись врача (логин/пароль) через админку.

## User Journey
1. **Выдача доступа**: Администратор открывает `/admin`, переходит в «Настройка врачей» и создает профиль (логин/пароль).
2. **Авторизация**: Врач переходит на `/doctors` и вводит свои данные для входа.
3. **Фильтрация**: Врач видит все сессии. Он вводит свою фамилию в фильтр по врачам и выбирает нужный диапазон дат. Браузер запоминает настройки фильтра для последующих визитов.
4. **Работа с сессией**: Врач находит строку нужного пациента, видит время начала и конца (в читаемом виде), длительность в минутах. Он нажимает «Просмотр» для открытия PDF в соседней вкладке или «Скачать PDF» для загрузки файла на компьютер.

## Functional Requirements

### Must Have (P0)
- **Модель базы данных для врачей**: Таблица для хранения логинов, хэшей паролей и, возможно, ФИО врача.
- **Интеграция SQLAdmin**: Раздел CRUD для управления моделью врачей администратором.
- **API авторизации для портала врачей**: Эндпоинты `/api/v1/doctors/login` и проверки сессии (с выдачей JWT токена).
- **Синхронизация с Битрикс24**: При получении или обновлении сделки бэкенд должен считывать одно из пользовательских полей, в зависимости от ID воронки:
  - Воронка `0`: `UF_CRM_1665032105080`
  - Воронка `1`: `UF_CRM_1688542532`
  - Воронка `3`: `UF_CRM_1616736315899`
  И сохранять это значение (ФИО врача) в запись сессии в БД БД PostgreSQL.
- **API списка сессий**: Выдача сессий с полями: `patient_name`, `doctor_name`, `start_time`, `end_time`, `duration_minutes`, `pdf_url`.
- **Frontend SPA (Лендинг)**:
  - Форма логина.
  - Главная страница с таблицей (колонки: ФИО пациента, ФИО врача, Время начала и конца, Длительность, Кнопки PDF).
  - Панель фильтров: Поиск по врачу (строка) и Фильтр по дате (выбор периода).
  - Логика сохранения фильтров (localStorage Zustand store).

### Should Have (P1)
- Красивый, современный, визуально приятный дизайн таблицы и кнопок действий, превосходящий текущую системную админку (TailwindCSS).

## Technical Architecture

### Data Model (Изменения)
- **New Entity: `DoctorUser`**: `id`, `username` (логин), `hashed_password`, `created_at`.
- **Update Entity: `SurveySession`** (или аналогичная таблица): Добавить колонку `doctor_name` (string, nullable), в которую будет записываться ФИО врача из Битрикса.

### System Components
- **Backend (FastAPI)**: Новые эндпоинты в `api/v1/endpoints/` (например, `doctor_portal.py`). Доработки в `bitrix24.py` для парсинга полей.
- **Frontend (React/Vite)**: Новые страницы в `src/pages/doctors/`, новый стор `useDoctorStore.ts`.
- **Reverse Proxy (Nginx)**: Убедиться, что `/doctors` корректно обслуживается через SPA-маршрутизацию `vite.config.ts` и `nginx`.

### Security Model
- JWT-авторизация (отдельный скоуп или роль `doctor`, чтобы врачи не могли дергать API основной админки).
- Хэширование паролей стандартным для проекта алгоритмом (bcrypt/passlib).
- Врачи видят все данные сессий (нет row-level security по врачам), но доступ к лендингу закрыт без авторизации.

## Open Questions for Implementation
- Будет ли история старых сделок (уже завершенных сессий) обновлена (ретроспективно подтянуто ФИО врача из Битрикса)? Если да, потребуется разовый скрипт миграции. Иначе в старых сессиях колонка "Врач" будет пустой.
- Как точно хранятся ФИО пациента в текущей БД? (Нужно убедиться, что они доступны для быстрой выдачи в табличном API без тяжелых джоинов или парсинга JSON для каждой строки).
