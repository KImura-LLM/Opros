# ТЗ и технический TODO: вкладки портала врачей с маршрутизацией по воронкам Bitrix24

## 1. Краткое описание задачи

Нужно доработать существующий портал врачей `/doctors`, чтобы входящие сессии опросников автоматически раскладывались по вкладкам в зависимости от воронки Bitrix24, из которой пришла сделка.

На странице врача должны появиться вкладки:

- `Новосибирск`
- `Кемерово`
- `Ярославль`
- `Тест`

При этом вкладка `Тест` должна быть скрыта для большинства пользователей и показываться только тем врачам или сотрудникам, у которых в учетной записи включен специальный флаг доступа.

Документ ниже фиксирует согласованные бизнес-правила и содержит технический план реализации на базе уже существующего doctor portal.

---

## 2. Подтвержденные решения

### 2.1. Маршрутизация по воронкам

Используется `CATEGORY_ID` сделки Bitrix24.

Правила распределения:

| ID воронки Bitrix24 | Вкладка в портале |
|---|---|
| `0` | `Новосибирск` |
| `1` | `Кемерово` |
| `3` | `Ярославль` |
| Любой другой ID | `Тест` |
| `CATEGORY_ID` отсутствует / не удалось определить | `Тест` |

### 2.2. Поведение вкладки `Тест`

- Это улавливающая вкладка для всех нестандартных, служебных, тестовых и неописанных воронок.
- Данные из `Тест` не должны смешиваться с реальными клиниками.
- Пользователь без разрешения не должен:
  - видеть саму вкладку,
  - открыть ее вручную через интерфейс,
  - получить данные этой вкладки через API.

### 2.3. Доступы

- Флаг доступа добавляется в существующую модель пользователей врачебного портала.
- Название флага в админке: `Доступ к тестовой вкладке`.
- Пока что все врачи видят все сессии, кроме ограничения на вкладку `Тест`.

### 2.4. Старые записи

- Разметка нужна не только для новых, но и для старых сессий.
- Для старых записей требуется отдельный процесс дозаполнения данных.

### 2.5. Сортировка

- Внутри вкладки записи сортируются по дате завершения опроса.
- Самые новые записи должны быть сверху.

---

## 3. Почему выбран такой подход

Чтобы не нагружать страницу лишними данными и не тащить в браузер весь список сессий сразу, фильтрацию по вкладкам лучше делать на backend.

Это дает сразу несколько плюсов:

- в браузер уходит только нужная вкладка;
- проще скрыть `Тест` не только визуально, но и реально по правам;
- меньше риск случайно показать лишние данные;
- легче масштабировать портал, если объем сессий вырастет.

Итоговое решение:

- frontend хранит выбранную вкладку;
- backend принимает активную вкладку как параметр;
- backend сам проверяет права пользователя и возвращает только допустимые данные;
- у каждой сессии в базе хранится уже нормализованный признак вкладки, чтобы не пересчитывать его на каждый запрос.

## 4. Подробное ТЗ

## 4.1. Текущая база, от которой отталкиваемся

В проекте уже существует основа портала врачей:

- модель врача: `backend/app/models/doctor_user.py`
- API портала: `backend/app/api/v1/endpoints/doctors.py`
- страница портала: `frontend/src/pages/doctors/DoctorPortalPage.tsx`
- интерфейс таблицы: `frontend/src/pages/doctors/DoctorDashboard.tsx`
- админка врачей: `backend/app/admin/doctor_view.py`

Значит, задача не про создание нового портала, а про расширение уже работающего раздела.

## 4.2. Бизнес-цель

Сделать так, чтобы врач после входа видел сессии, уже разложенные по клиникам, без ручной сортировки и без риска, что тестовые сделки попадут в рабочие вкладки.

## 4.3. Визуальная структура страницы врача

На странице `/doctors` нужно добавить панель вкладок.

Порядок вкладок:

1. `Новосибирск`
2. `Кемерово`
3. `Ярославль`
4. `Тест` — только при наличии права доступа

Требования к поведению:

- по умолчанию открывается первая доступная вкладка;
- если у пользователя нет доступа к `Тест`, вкладка не отображается вообще;
- переключение вкладок обновляет список сессий без перезагрузки страницы;
- текущие фильтры по врачу и датам продолжают работать внутри активной вкладки;
- выбранную вкладку можно сохранять в клиентском состоянии, но backend все равно остается главным источником прав доступа.

## 4.4. Логика распределения данных

Каждая сессия должна быть привязана ровно к одной вкладке.

Правило:

- если `CATEGORY_ID = 0`, сессия относится к `Новосибирск`;
- если `CATEGORY_ID = 1`, сессия относится к `Кемерово`;
- если `CATEGORY_ID = 3`, сессия относится к `Ярославль`;
- если `CATEGORY_ID` любой другой, сессия относится к `Тест`;
- если `CATEGORY_ID` отсутствует, пустой, поврежден или не удалось получить сделку Bitrix, сессия также относится к `Тест`.

Важно:

- одна и та же сессия не может одновременно лежать в двух вкладках;
- логика должна быть одинаковой для новых и старых записей;
- backend не должен полагаться только на frontend-фильтрацию.

## 4.5. Какие данные нужно хранить у сессии

Чтобы не ходить в Bitrix24 на каждый просмотр списка, у сессии нужно хранить нормализованный результат маршрутизации.

Рекомендуемые поля в `SurveySession`:

- `bitrix_category_id` — исходный ID воронки из Bitrix24, nullable integer;
- `portal_clinic_bucket` — нормализованное значение вкладки.

Допустимые значения `portal_clinic_bucket`:

- `novosibirsk`
- `kemerovo`
- `yaroslavl`
- `test`

Причина, почему нужно хранить оба значения:

- `bitrix_category_id` полезен для аудита и последующих проверок;
- `portal_clinic_bucket` нужен для быстрых выборок и простой логики на backend/frontend.

## 4.6. Обновление новых данных

Когда создается или обновляется сессия, связанная со сделкой Bitrix24:

1. система получает `CATEGORY_ID` сделки;
2. сохраняет его в `bitrix_category_id`;
3. по таблице правил вычисляет `portal_clinic_bucket`;
4. сохраняет bucket в сессию;
5. при необходимости обновляет старое пустое или некорректное значение.

Эта логика должна работать в тех местах, где уже идет привязка doctor portal к Bitrix24:

- при завершении опроса;
- при обработке webhook;
- при дополнительных доборах данных по сделке, если они уже реализованы в текущем коде.

## 4.7. Обработка старых записей

Для уже существующих сессий нужно выполнить разовый backfill.

Требования:

- пройти по старым завершенным сессиям;
- для каждой записи по `lead_id` запросить сделку Bitrix24;
- определить `CATEGORY_ID`;
- заполнить `bitrix_category_id`;
- вычислить и сохранить `portal_clinic_bucket`;
- если `lead_id` отсутствует или сделку получить невозможно, отнести запись в `test`.

Требования к надежности:

- backfill должен быть идемпотентным, чтобы его можно было безопасно запускать повторно;
- изменения должны быть пакетными, без переполнения памяти;
- ошибки по отдельным записям не должны останавливать весь процесс;
- нужен понятный лог: сколько записей обновлено, сколько ушло в `test`, сколько не удалось обработать.

## 4.8. API портала врачей

Существующий endpoint списка сессий нужно расширить.

Новые требования:

- backend принимает параметр вкладки, например `tab` или `clinic_bucket`;
- backend возвращает только записи активной вкладки;
- backend сортирует по `completed_at desc`;
- backend применяет существующие фильтры по врачу и диапазону дат поверх активной вкладки;
- backend не возвращает `test`, если у врача нет соответствующего флага.

Рекомендуемое поведение:

- если пользователь просит `test`, но флага нет, backend возвращает `403`;
- если передан неизвестный tab, backend возвращает `422` или нормализованную ошибку валидации;
- endpoint `/me` должен отдавать признак, может ли пользователь видеть `Тест`.

## 4.9. Право доступа к вкладке `Тест`

В модель `DoctorUser` добавляется булево поле:

- внутреннее имя: например `can_view_test_tab`
- отображаемое имя в админке: `Доступ к тестовой вкладке`

Требования:

- по умолчанию значение `false`;
- поле редактируется в SQLAdmin;
- логика проверки права должна быть и в backend, и в интерфейсе;
- скрытие вкладки только на frontend не считается достаточной защитой.

## 4.10. Изменения в интерфейсе врача

На frontend необходимо:

- добавить вкладки над таблицей;
- получать список доступных вкладок на основе ответа `/me`;
- не рендерить `Тест`, если право не выдано;
- при смене вкладки заново запрашивать список сессий;
- хранить выбранную вкладку в store;
- оставить текущие фильтры по датам и врачу;
- сортировку на frontend не дублировать, если она уже делается на backend.

Желаемое поведение по UX:

- вкладки переключаются быстро и предсказуемо;
- состояние активной вкладки не сбрасывается без необходимости;
- если у пользователя исчез доступ к `Тест`, сохраненное ранее состояние не должно ломать страницу: нужно автоматически переключать его на первую доступную рабочую вкладку.

## 4.11. Фильтрация и сортировка

Порядок применения правил:

1. ограничение по правам пользователя;
2. выбор активной вкладки;
3. фильтр по дате;
4. фильтр по врачу;
5. сортировка по завершению опроса, новые сверху.

Для дат предпочтительно использовать `completed_at`, потому что пользователь просил сортировку именно по завершению опроса.

Если текущая логика фильтрации привязана к `started_at`, ее нужно привести к единому правилу и документированно перевести на `completed_at`, чтобы не было расхождения между сортировкой и фильтром периода.

## 4.12. Ошибки и крайние случаи

Система должна устойчиво обрабатывать такие сценарии:

- у сессии нет `lead_id`;
- сделка Bitrix24 удалена или недоступна;
- в сделке нет `CATEGORY_ID`;
- пришел незнакомый ID воронки;
- у врача убрали доступ к `Тест` во время действующей сессии;
- старые записи еще не полностью размечены.

Во всех неоднозначных случаях данные должны уходить в `Тест`, а не в рабочие вкладки клиник.

## 4.13. Безопасность и приватность

- Пользователь без флага не должен получать доступ к данным вкладки `Тест` даже через прямой запрос к API.
- Логика доступа должна проверяться на backend по текущему пользователю.
- Нельзя хранить в клиентском постоянном хранилище ничего, что связано с медицинскими ответами пациентов.
- Хранение выбранной вкладки и служебных фильтров допустимо, так как это не чувствительные медицинские данные.

## 4.14. Критерии приемки

Функция считается реализованной, если выполнено все ниже:

1. После входа врач видит вкладки `Новосибирск`, `Кемерово`, `Ярославль`.
2. Вкладка `Тест` показывается только врачам с флагом `Доступ к тестовой вкладке`.
3. Сделки из воронок `0`, `1`, `3` всегда попадают строго в свои вкладки.
4. Любые другие воронки, а также отсутствие `CATEGORY_ID`, всегда попадают строго в `Тест`.
5. Записи из `Тест` не видны пользователям без разрешения ни в интерфейсе, ни через API.
6. Старые записи после backfill тоже распределены по правилам.
7. Список в каждой вкладке отсортирован по завершению опроса, новые сверху.
8. Фильтры по врачу и датам продолжают работать.
9. После изменения прав в админке backend корректно применяет доступ.
10. Решение не ломает текущую выдачу PDF, предпросмотр и авторизацию портала врачей.

## 4.15. Что не входит в эту задачу

Пока не делаем:

- ограничение видимости сессий по конкретному врачу;
- отдельные права по вкладкам реальных клиник;
- редактирование правил маршрутизации из админки;
- автоматическую синхронизацию исторических записей по расписанию, если достаточно разового backfill;
- переработку визуального стиля всего doctor portal за пределами вкладок и связанных фильтров.

## 5.14. Риски, которые нельзя пропустить

- Если сделать скрытие `Тест` только на frontend, данные можно будет получить прямым API-запросом.
- Если не сохранить bucket в БД, портал будет лишний раз зависеть от Bitrix24 и станет медленнее.
- Если backfill выполнить без логов и повторного запуска, будет трудно понять, какие старые записи остались неразмеченными.
- Если оставить фильтр по `started_at`, а сортировку сделать по `completed_at`, пользователь увидит путаницу в периодах.
- Если незнакомые воронки не отправлять в `Тест` по умолчанию, есть риск смешать тестовые и рабочие данные.

## 6. Итог для старта разработки

Основная идея реализации:

- хранить у каждой сессии исходный ID воронки и готовый bucket вкладки;
- фильтровать вкладки на backend, а не в браузере;
- скрывать `Тест` и в интерфейсе, и на уровне API;
- добавить флаг доступа в существующую модель `DoctorUser`;
- разово доразметить старые записи через backfill.

Это решение минимизирует нагрузку, не ухудшает пользовательский опыт и снижает риск смешивания тестовых и реальных данных.
